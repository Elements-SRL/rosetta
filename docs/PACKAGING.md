# Building the Windows installer

`e384c` is now linked statically into `rosetta.exe` (see `e384_rust`'s
`build.rs`), so `e384c.dll` is no longer part of the installer. `rosetta`
still links dynamically against 3 vendor DLLs (`FTD2XX.dll`, `MPSSE.dll`,
`okFrontPanel.dll`) and needs the Microsoft VC++ runtime at run time.
Committed copies of these live under `vendor\dlls\` and `vendor\redist\` so the
installer build is reproducible from a clean clone, with no dependency on the
sibling `e384c` repo or any developer-machine paths.

The release process produces a single `RosettaSetup.exe` that:
1. Silently installs the VC++ Redistributable (x64) if it isn't already present.
2. Installs `rosetta.exe` plus all 3 DLLs into the same folder under
   `Program Files\rosetta\bin\`.

## One-time setup

- Install the WiX v3.14 toolset (`candle.exe`/`light.exe`). Either install it
  system-wide, or download the binaries zip from
  https://github.com/wixtoolset/wix3/releases/tag/wix3141rtm and unzip it
  somewhere (e.g. `C:\wix-toolset\bin`).
- `cargo install cargo-wix`

## Updating the vendored DLLs / VC++ redistributable

- `vendor\dlls\*.dll` — replace with fresh copies from the FTDI/Opal Kelly
  SDK/driver install whenever the vendor SDK is updated. `e384c` itself is
  linked statically (via `E384C_LIB_DIR`/`E384COMMLIB_LIB_DIR`, see
  `e384_rust`'s `build.rs`), so there's no `e384c.dll` to vendor here.
- `vendor\redist\vc_redist.x64.exe` — Microsoft's official, redistributable
  installer. Refresh periodically from https://aka.ms/vs/17/release/vc_redist.x64.exe.

## Release build steps

### Quick way: run the script

From the repo root, `build-installer.ps1` runs all three steps below for you. It
reads the version from `Cargo.toml`, finds the WiX toolset, checks each step
succeeded, and prints the final installer path:

```powershell
# WiX on PATH, or set $env:WIX_BIN first
.\build-installer.ps1

# or point it at the toolset explicitly
.\build-installer.ps1 -WixBin "C:\Users\lross\wix-toolset\bin"
```

Pass `-Version <x.y.z>` only if you need to override the value from `Cargo.toml`.

### Manual steps

Run from the repo root in PowerShell. First set `$WixBin` to wherever
`candle.exe`/`light.exe` live. `$Version` is read automatically from
`Cargo.toml` so it always matches the MSI that `cargo wix` produces — you never
edit it by hand. These are PowerShell variables, not literal text to paste into
the commands below (don't type angle brackets like `<WIX_BIN>` directly into
PowerShell — it parses a bare `<...>` as a redirection operator and errors
out):

```powershell
$WixBin = "C:\Users\lross\wix-toolset\bin"

# Read the package version straight from Cargo.toml (authoritative, via cargo).
$Version = (cargo metadata --no-deps --format-version 1 | ConvertFrom-Json).packages `
    | Where-Object { $_.name -eq 'rosetta' } | Select-Object -ExpandProperty version

cargo wix -b $WixBin
& "$WixBin\candle.exe" installer\bundle.wxs -ext WixBalExtension -ext WixUtilExtension "-dVersion=$Version" -out target\wix\bundle\
& "$WixBin\light.exe" target\wix\bundle\bundle.wixobj -ext WixBalExtension -ext WixUtilExtension -out target\wix\RosettaSetup.exe
```

`cargo wix` already runs `cargo build --release` for you, so there's no need
to invoke it separately beforehand.

> Why derive it? `cargo wix` always names its MSI `rosetta-<Cargo.toml
> version>-x86_64.msi`, and `bundle.wxs` looks for exactly that filename via
> `$(var.Version)`. Reading `$Version` from `Cargo.toml` keeps the two in sync
> automatically — bump the version in `Cargo.toml` and the installer follows.
>
> If you'd rather not shell out to `cargo`, this regex reads the same value
> straight from the file (the package `version` is the first `version = "..."`
> line):
>
> ```powershell
> $Version = (Select-String -Path Cargo.toml -Pattern '^version\s*=\s*"(.*)"').Matches[0].Groups[1].Value
> ```

The final distributable is `target\wix\RosettaSetup.exe` — hand this single file to
end users.

Note: `bundle.wxs` deliberately lives in `installer\`, not `wix\` — `cargo wix`
auto-compiles every `.wxs` file it finds directly in `wix\`, and having both the
`<Product>` (MSI) and `<Bundle>` definitions in that folder makes `cargo wix` try to
link them together and fail with `LGHT0089: Multiple entry sections`. Keeping
`bundle.wxs` in a separate folder avoids that; its relative paths (e.g.
`vendor\dlls\...`, `target\wix\rosetta-<VERSION>-x86_64.msi`) are resolved against the
current working directory when you invoke `candle`/`light`, not against the `.wxs`
file's own location, so running the commands above from the repo root just works.

## Files

- `wix\main.wxs` — MSI definition (generated by `cargo wix init`, then hand-edited to
  add the 3 vendor DLLs alongside `rosetta.exe`).
- `installer\bundle.wxs` — WiX Bundle (Burn) chaining the VC++ redistributable and the
  MSI into one `setup.exe`.
